# Agent Notes: The Dual-State Recon Engine (Streaming vs. Polling)

**Date:** 2026-02-25
**Context:** Resolving mid-turn UI truncations, "phantom" duplicate threads, and multi-UUID identity drift between the Web View UI and the underlying CLI providers (Codex, Claude, Gemini).

---

## 1. The Core Architectural Tension

The Web View acts as a stateful middleman between a fast React client and slow, varying CLI binaries. 
The system has two fundamentally different sources of truth that are constantly racing each other:

1.  **The Live Runtime (WebSocket Stream):** Provided by `@nbardy/agent-cli`. As a CLI runs, it emits high-frequency `text.delta`, `tool.use`, and `turn.started` events.
2.  **The Persisted Artifacts (Disk Poller):** Provided by the `server/src/adapters/` (e.g., `jsonl.ts`). A background `setInterval` wakes up every 5 seconds to scan the user's hard drive for changes to `~/.codex/sessions` or `~/.claude/projects`.

### The Problem
If the UI tries to blindly merge these two sources, state becomes corrupted.
*   **Duplicate Threads:** The UI creates an optimistic UUID (`d97e...`). The CLI starts and creates a file named `019c...`. The 5s poller finds the file, doesn't realize it belongs to the UI's UUID, and imports it as a brand-new, duplicate conversation in the sidebar.
*   **Mid-Turn Truncation:** The UI manually stitches streaming `text.delta` chunks into the `messages` array. If the server crashes mid-turn, the UI state is permanently corrupted because it drifted from the actual disk file.

---

## 2. The Solution: Dual-State Reconciliation

We implemented a strict boundary between "In-Flight" data and "Committed" data, governed by the following rules.

### Rule 1: Explicit Dual-Identity Contract
We cannot overload a single `id` field.
*   **`conversation.id`:** A permanent, client-generated UUID. This is the primary key for the UI. It never changes.
*   **`conversation.sessionId`:** The native ID generated by the provider (e.g., the Codex thread ID).
*   **The Handshake:** When the UI creates a chat, `sessionId` is undefined. When the server spawns the CLI and receives `session.started`, it explicitly broadcasts a **`session_bound`** event to the client. The UI now knows its local UUID is tied to that specific disk artifact.

### Rule 2: Aggressive Epoch Sync (Killing "Ghost" Threads)
Previously, the React client used an intricate merge strategy during `case 'init':` to preserve its local state when reconnecting to the WebSocket. This caused stale "ghost" threads to survive forever.
*   **The Fix:** The Server is the absolute Epoch. On `init`, the client performs a **total wipe and replace** of its `conversationsAtom`. It only preserves optimistic stubs (chats the user *just* hit enter on that haven't reached the server yet).

### Rule 3: The "Bootstrap" Interceptor (Killing Duplicates)
When the 5s disk poller (`pollForChanges`) discovers a new `.jsonl` file, it must not blindly import it.
*   **The Fix:** It runs `findBootstrapMatch()`. It searches the in-memory Map for any conversation where the `sessionId` is unassigned, the `workingDirectory` matches, and the first user prompt is identical. If it finds one, it silently aliases the disk file to the UI's UUID, preventing a duplicate from ever hitting the UI.

### Rule 4: In-Flight Isolation (Killing Truncation)
The UI `messages` array is strictly reserved for **Committed Ground Truth** loaded from disk. 
*   **The Fix:** `text.delta` streams are written into a highly volatile, transient buffer (`streamingContentAtom`). 
*   When the server broadcasts `message_complete`, the UI **drops the volatile buffer entirely**. It does not attempt to manually merge the string. Instead, it relies on the server immediately broadcasting a `conversations_updated` event containing the newly parsed Ground Truth from the disk adapter. 

---

## 3. Key Takeaways for Future Agents

*   **Never trust the UI UUID to exist on disk.** The disk adapters know nothing about the UI. They only know about provider session IDs.
*   **Do not filter the Sidebar on the Server.** We attempted to filter `conversations_updated` by the server's `cwd` to stop "phantom splits" (external terminal sessions appearing in the UI). This broke the UI's ability to show multiple projects (like `font_maker` and `llm_trader` simultaneously). The Server must send all data; the UI handles its own display logic.
*   **`@nbardy/agent-cli` is Solid:** If streaming seems broken, the bug is almost certainly in how `server.ts` or `actions.ts` is consuming the `UnifiedAgentEvent` stream, not in the CLI parser itself.
